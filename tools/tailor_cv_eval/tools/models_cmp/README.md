# Model Comparison Framework

This directory contains the implementations for comparing and evaluating CVs generated by different local Ollama models against a reference ground truth. By running these benchmarks, you can understand how well different models (e.g., `llama3.1:8b`, `deepseek-r1:7b`, `gemma4:e2b`) perform the task of tailoring a CV.

---

## Overview of Comparison Tools

We support **7 different evaluation engines** to assess the generated CVs. Each engine has unique strengths and focuses:

| Tool | Primary Use Case | Output Format | Evaluation Method | Key Strength |
| :--- | :--- | :--- | :--- | :--- |
| **DeepEval** | Factual Correctness / CI Testing | Pytest Console & HTML | G-Eval model-graded check | Integrates with Pytest, great for CI/CD |
| **Promptfoo** | Prompt tuning / Embedding comparison | CLI & Interactive React Web Dashboard | Embeddings cosine similarity, LLM rubric, ROUGE | Extremely visual, tracks multiple metrics |
| **LlamaIndex** | RAG / Agentic evaluation | CLI / Console outputs | CorrectnessEvaluator (model-graded) | Very simple, built-in correctness checking |
| **Ragas** | RAG system auditing | CLI / Console outputs | Faithfulness, Answer Relevance metrics | Focuses on retrieval consistency and hallucinations |
| **TruLens** | Logging & tracing evaluations | SQLite Database & TruLens Dashboard | LLM-graded numeric score logged to DB | Excellent for tracking runs and trends over time |
| **Giskard** | Model validation / Quality checks | CLI / Console outputs | Model-graded functional test suite | Structured datasets, easy to define assertion suites |
| **Text-Grad** | Auto-differentiation of prompts / feedback | CLI / Console outputs | Textual Backpropagation (Loss & Gradients) | Provides actionable feedback and iterative optimization |

---

## Shared Architecture & Caching

To ensure fair comparison and save computing resources, the framework uses a shared orchestration flow:

1. **Prompt Generation**: A baseline prompt is generated and saved to `tmp/outputs/job1/llm_prompt.md`.
2. **Cached Generation**: Each model generates its tailored CV using the prompt. The outputs are saved to `tmp/outputs/job1/model_output/<model_name>_jd_cv.md`. If this file already exists, it is reused instead of calling Ollama again.
3. **Evaluation**: The tool compares the generated CV against the ground truth (`inputs/job1/gt.md`) using the evaluator model `EVAL_MODEL` (defaults to `llama3.1:8b`).

> [!TIP]
> If you make changes to a model's prompt or parameters and want to regenerate the CVs, make sure to delete the cached outputs in `tmp/outputs/job1/model_output/` or run `just clean`.

## Detailed Tool Configurations & Commands

### 1. DeepEval
DeepEval runs model-graded evaluations using the `GEval` metric. It checks whether the actual output is factually correct based on the expected output.
* **Commands**:
  * **Run**: Runs as part of `just test-jd` or directly via:
    `uv run deepeval test run tools/models_cmp/deepeval/test_eval.py -- --variant jd --html=tmp/outputs/job1/deepeval/report.html --self-contained-html`
  * **View Report**: `just deepeval-view`
* **Output**: A pytest console summary and an interactive HTML report saved at `tmp/outputs/job1/deepeval/report.html`.

### 2. Promptfoo
Promptfoo evaluates models against multiple assertions: semantic similarity (embeddings), factual consistency (LLM rubric), and lexical overlap (ROUGE-N).
* **Commands**:
  * **Run**: `just promptfoo-test-jd`
  * **View Dashboard**: `just promptfoo-view` (opens a web application on `http://localhost:15500`)
* **Output**: Configuration files are generated dynamically at `tmp/outputs/job1/promptfoo/promptfooconfig_jd.yaml`. View results in the web dashboard.

### 3. LlamaIndex
LlamaIndex uses the native `CorrectnessEvaluator` to compute a correctness score from `1.0` to `5.0` along with a detailed textual reasoning.
* **Commands**:
  * **Run**: `just llamaindex-test-jd`
* **Output**: Printed to console, and the generated CVs are saved to `tmp/outputs/job1/llamaindex/`.

### 4. Ragas
Ragas calculates `faithfulness` (factuality) and `answer_relevance` scores (ranging from `0.0` to `1.0`) by preparing a dataset format.
* **Commands**:
  * **Run**: `just ragas-test-jd`
* **Output**: Printed to console, and the generated CVs are saved to `tmp/outputs/job1/ragas/`.

### 5. TruLens
TruLens logs the execution details and custom correctness feedback functions. All records are persisted to a SQLite database.
* **Commands**:
  * **Run**: `just trulens-test-jd`
* **Output**: Records and evaluations are logged in `tmp/outputs/truelens/default.sqlite`.

### 6. Giskard
Giskard wraps the evaluation in a Giskard Dataset and executes a custom unit-test suite checking for factual correctness.
* **Commands**:
  * **Run**: `just giskard-test-jd`
* **Output**: Printed to console, and the generated CVs are saved to `tmp/outputs/job1/giskard/`.

### 7. Text-Grad
Text-Grad treats prompts and outputs as variables, performing a backpropagation-like pass. It generates a detailed "Loss Evaluation" and "Textual Feedback for Improvement" (textual gradients) pointing out missing sections, formatting issues, or factual inconsistencies.
* **Commands**:
  * **Run**: `just textgrad-test-jd`
* **Output**: Detailed textual feedback printed to console, and the generated CVs are saved to `tmp/outputs/job1/textgrad/`.

## How to Compare Models

### Step 1: Set Up Environment & Models
Ensure you have all the comparison models downloaded in Ollama:
```bash
# Pull the candidate models
ollama pull gemma4:e2b
ollama pull deepseek-r1:7b
ollama pull qwen2.5:7b
ollama pull llama3.1:8b
ollama pull deepseek-r1:1.5b
```

### Step 2: Generate the Evaluation Prompt
Generate the prompt that will be fed into the candidate models:
```bash
just generate-prompt-for-jd      # For job description tailoring
```

### Step 3: Run the Evaluations

#### Option A: Run a single tool
If you want to focus on a specific evaluation framework, run its target:
```bash
just textgrad-test-jd
```

#### Option B: Run all tools
To execute all 7 evaluation frameworks sequentially, run:
```bash
just test-jd
```

---

## Customization

You can customize the evaluator model, candidate models, and their custom options directly in `config.yaml` at the root of the framework.
